# 资产中心 (Asset Hub) 模块深度分析报告

> 分析日期: 2026-04-16 | 项目版本: v0.4.1

---

## 一、模块概述

资产中心是 waoowaoo 的全局素材管理模块，用于管理跨项目复用的角色、场景、道具和音色资产。用户可在此创建、编辑、生成图片，并将资产复制到具体项目中使用。

**核心文件**:
- 页面入口: `src/app/[locale]/workspace/asset-hub/page.tsx` (662 行)
- 资产网格: `src/app/[locale]/workspace/asset-hub/components/AssetGrid.tsx` (460 行)
- 组件目录: `src/app/[locale]/workspace/asset-hub/components/` (14 个组件文件)
- 共享组件: `src/components/shared/assets/` (创建/编辑 Modal)
- Mutation Hooks: `src/lib/query/mutations/asset-hub-*.ts` (6 个文件, 33 个 mutation)
- API 路由: `src/app/api/asset-hub/` (20 个子目录) + `src/app/api/assets/` (统一资产操作)

## 二、UI/UX 问题

### 2.1 空状态下的冗余创建入口

**现象**: 当资产中心没有任何资产时，页面同时展示三个创建入口：
1. 右上角 `+ 新建资产` 下拉按钮 (始终可见)
2. 空白区域中央一个**不可点击的灰色加号图标** (纯装饰)
3. 空白区域底部又一个 `+ 新建资产` 下拉按钮

**根因** (`AssetGrid.tsx:344-360`):
```jsx
{isEmpty ? (
  <div className="glass-surface rounded-xl p-12 text-center">
    {/* 纯装饰加号，无 onClick */}
    <div className="w-16 h-16 mx-auto mb-4 rounded-full ...">
      <PlusIcon className="w-8 h-8 text-[var(--glass-text-tertiary)]" />
    </div>
    <p>暂无资产</p>
    <p>点击上方按钮添加角色或场景</p>
    {/* 第二个新建资产按钮 */}
    <AddAssetDropdown ... />
  </div>
) : ...}
```

**问题**:
- 灰色加号图标看起来像可点击按钮，但实际无交互，误导用户
- 提示文案"点击上方按钮"与下方又放了一个按钮自相矛盾
- 两个 `+ 新建资产` 按钮功能完全相同，造成视觉噪音

**建议**:
- 方案 A: 空状态只保留中央区域的创建按钮，移除右上角按钮（空状态下无需顶部操作栏）
- 方案 B: 将灰色加号改为可点击的主操作按钮，移除底部的下拉按钮，保留右上角按钮
- 无论哪种方案，都应消除"不可点击的加号"这个反直觉元素

### 2.2 描述文案不准确

空状态提示"点击上方按钮添加角色或场景"，但实际支持四种资产类型（角色、场景、道具、音色），文案未覆盖道具和音色。

### 2.3 筛选 Tab 下的创建入口不联动

**现象**: 用户切换到具体类别 Tab（如"场景"）后，右上角的"新建资产"按钮仍然是全局的四类下拉菜单，需要在二级菜单中再次选择"新建场景"。没有根据当前 Tab 自动聚焦到对应类别。

**根因** (`AssetGrid.tsx:35-122`): `AddAssetDropdown` 组件不接收当前 filter 状态，始终展示完整的四类菜单。

**建议**: 当用户处于具体类别 Tab 时，"新建资产"按钮应直接变为"新建场景"（单一操作），或至少将当前类别置顶/高亮。

### 2.4 筛选 Tab 空状态文案通用化，无类别针对性

**现象**: 当有资产但当前筛选 Tab 下无结果时（如切到"道具"Tab 但没有道具），显示的是通用文案"点击新建资产添加资产" (`filteredEmptyHint`)，而非具体的"暂无道具，点击新建道具"。

**根因** (`AssetGrid.tsx:361-366`, `messages/zh/assetHub.json:25`):
```jsx
: visibleAssetCount === 0 ? (
    <div className="flex min-h-[320px] items-center justify-center">
        <p>{t('filteredEmptyHint')}</p>  {/* "点击新建资产添加资产" — 通用文案 */}
    </div>
)
```

空状态区域也没有提供快速创建按钮，用户必须回到右上角下拉菜单操作。

**建议**: 根据当前 `filter` 值动态生成文案和创建按钮，如 `filter === 'prop'` 时显示"暂无道具" + "新建道具"按钮。

### 2.5 图片数量选择器使用原生 HTML `<select>`

**现象**: 角色/场景卡片上的图片生成数量下拉框是浏览器原生 `<select>` 元素，与整体 Glass 设计系统风格不一致。

**根因** (`ImageGenerationInlineCountButton.tsx:92-104, 141-153`):
```jsx
<select
  value={String(value)}
  onChange={(event) => onValueChange(Number(event.target.value))}
  className={`${selectClassName} ${selectStateClassName}`.trim()}
>
  {options.map((option) => (
    <option key={option} value={option} className="text-black">{option}</option>
  ))}
</select>
```

`<select>` 元素没有自定义样式，调用方也不传 `selectClassName`，渲染为浏览器默认外观。`<option>` 只设了 `text-black`，下拉面板完全是原生样式。

**建议**: 替换为自定义的 Glass 风格下拉组件（如项目中已有的 `SegmentedControl` 或 `ui/primitives` 中的组件），或至少为 `<select>` 添加 `appearance-none` + 自定义样式覆盖。

### 2.6 "确定选择方案 X" 按钮位置与对齐

**现象**: 多图选择模式下，"确定选择方案 X" 按钮位于图片网格下方右对齐 (`flex justify-end`)，但与上方图片网格、下方音色设置区域之间的间距和视觉层级不够清晰，容易被忽略或与其他操作混淆。

**根因** (`CharacterCard.tsx:336-348`):
```jsx
{effectiveSelectedIndex !== null && (
    <div className="mt-4 flex justify-end">
        <button onClick={handleConfirmSelection} ...>
            确定选择方案 {effectiveSelectedIndex + 1}
        </button>
    </div>
)}
```

按钮仅在选中某张图片后才出现，但出现位置紧贴图片网格底部，与下方的 VoiceSettings 组件之间没有视觉分隔。在卡片内容较多时，按钮容易淹没在其他元素中。

**建议**:
- 将确认按钮放在图片网格区域内部（如浮动在底部），或使用更醒目的样式（如全宽按钮）
- 增加与下方 VoiceSettings 的间距或分隔线
- 考虑在图片选择模式下隐藏不相关的操作区域（如音色设置），减少视觉干扰

## 三、角色创建与编辑流程问题

### 3.1 参考图创建角色后无法编辑属性

**现象**: 通过参考图创建角色后，用户只能编辑"名称"和"形象描述"，无法编辑角色档案属性（性别、年龄、性格、服装等级等）。

**根因**: 资产中心完全没有角色档案 (Profile) 编辑功能。
- `CharacterProfileCard.tsx` 和 `CharacterProfileDialog.tsx` 仅存在于项目工作区 (`novel-promotion/components/assets/`)
- 资产中心的 `CharacterEditModal` (306 行独立实现) 只提供名称和描述编辑
- `GlobalCharacter` 表有 `profileData` 和 `profileConfirmed` 字段，但资产中心 UI 从未使用

**影响**: 用户在资产中心创建的角色缺少结构化档案数据，复制到项目后仍需重新配置。

**建议**: 将 `CharacterProfileDialog` 抽取为共享组件，在资产中心的角色卡片上增加"编辑档案"入口。

### 3.2 参考图模式流程不完整

**创建流程缺陷**:

1. **FAL API Key 硬依赖** (已修复): `reference-to-character.ts:212` 原先无条件要求 FAL API Key，即使用户配置的图片模型不是 FAL。非 FAL 用户无法使用参考图功能。

2. **参考图轮询逻辑泄漏**: `reference-to-character.ts` 是唯一一个绕过任务系统异步轮询框架、在 handler 内部自己手动做 FAL 轮询的地方 (第 68-83 行)。其他所有 image handler 都通过 `async-poll.ts` 统一处理。这导致：
   - handler 需要关心具体 provider 的实现细节
   - 如果 provider 不是 FAL 但返回了异步结果，轮询逻辑无法处理

3. **提取描述后无法编辑再生成**: 参考图"先提取描述"模式下，提取完成后描述会填入文本框，但用户编辑描述后点击生成，`customDescription` 参数会覆盖提取结果，而非与参考图结合使用。

### 3.3 `GlobalCharacter` 缺少 `introduction` 字段

`NovelPromotionCharacter` 有 `introduction` 字段用于存储角色介绍（身份、关系、称呼映射），但 `GlobalCharacter` 没有。全局角色复制到项目后，`introduction` 为空，需要用户重新填写。

## 四、缓存刷新逻辑问题

### 4.1 图片选择确认后 UI 不刷新 (P0)

**复现**: 生成多张角色/场景参考图 → 选择其中一张 → 点击"确认" → UI 不更新，需刷新页面。

**根因** (`asset-hub-character-mutations.ts:205-246`):

乐观更新 `applyCharacterSelection` 仅更新 `selectedIndex` 和 `imageUrl`，但后端确认时会：
- 删除所有非选中图片
- 将 `imageUrls` 缩减为单元素数组
- 将 `selectedIndex` 重置为 0
- 合并 `descriptions` 为单元素数组

乐观缓存与服务端状态严重不匹配。`onSettled` 中的 `void invalidateCharacters()` 是 fire-and-forget，可能未能可靠覆盖乐观数据。

**同样受影响**: `useSelectLocationImage` 存在完全相同的问题模式。

**对比项目工作区**: 项目工作区使用独立的 `useConfirmProjectCharacterSelection` mutation，`onSettled` 无条件 invalidate 且不做乐观更新。

**修复建议**:
- confirm 操作不做乐观更新，直接等 `invalidateCharacters()` 的 refetch 结果
- 或在 confirm 时让乐观更新模拟后端行为（缩减 imageUrls、重置 selectedIndex）
- 将 `void invalidateCharacters()` 改为 `await invalidateCharacters()`

### 4.2 AI 修改描述后缓存未刷新 (P1)

以下 mutation 在操作成功后**完全没有调用 `invalidateQueries`**:

| Mutation | 文件 | 影响 |
|----------|------|------|
| `useAiModifyCharacterDescription` | `asset-hub-update-mutations.ts` | AI 修改角色描述后 UI 不更新 |
| `useAiModifyLocationDescription` | `asset-hub-update-mutations.ts` | AI 修改场景描述后 UI 不更新 |
| `useAiModifyPropDescription` | `asset-hub-update-mutations.ts` | AI 修改道具描述后 UI 不更新 |
| `useAiDesignCharacter` | `asset-hub-creation-mutations.ts` | AI 设计角色描述后 UI 不更新 |
| `useAiDesignLocation` | `asset-hub-creation-mutations.ts` | AI 设计场景描述后 UI 不更新 |

注: AI Design 类 mutation 在创建 Modal 内使用，不直接影响列表缓存，但 AI Modify 类 mutation 在编辑 Modal 中使用，关闭 Modal 后列表数据可能过期。

### 4.3 缓存失效策略不一致

| 模式 | 使用位置 | 问题 |
|------|---------|------|
| `onSuccess: invalidate` | undo, upload, delete appearance, upload voice | 仅成功时刷新 |
| `onSettled: invalidate` | generate, modify, delete | 成功或失败都刷新 |
| `onSettled: conditional void invalidate` | select (confirm only) | 条件性 fire-and-forget |
| 无 invalidation | AI modify/design | 完全不刷新 |

**建议**: 统一使用 `onSettled: invalidate` 模式，确保无论成功失败都刷新缓存。

## 五、代码重复与架构问题

### 5.1 组件重复实现统计

**共享组件已存在但 Asset Hub 未使用的情况**:

| 组件 | 共享版本 (行数) | Asset Hub 独立版本 (行数) | 项目工作区版本 (行数) | Asset Hub 是否使用共享版本 |
|------|----------------|--------------------------|---------------------|-------------------------|
| CharacterEditModal | 372 | 306 | 64 (薄包装) | ❌ 完全独立实现 |
| LocationEditModal | 320 | 292 | 42 (薄包装) | ❌ 完全独立实现 |
| CharacterCard | — (无共享版) | 520 | 491 | — 两套独立实现 |
| LocationCard | — (无共享版) | 489 | 413 | — 两套独立实现 |
| VoiceSettings | — (无共享版) | 192 | 225 | — 两套独立实现 |

**重复代码总量**: Asset Hub 独立实现约 1799 行，其中 CharacterEditModal (306 行) 和 LocationEditModal (292 行) 可直接替换为共享组件的薄包装 (~50 行 × 2)，节省约 500 行。

### 5.2 CharacterEditModal 独立实现的具体差异

Asset Hub 版 (`asset-hub/components/CharacterEditModal.tsx`, 306 行):
- 不使用 `AiModifyDescriptionField` 共享组件，内联实现 AI 修改 UI
- 使用 `mutate()` + `onError` 回调，而非 `mutateAsync` + try-catch
- 不支持 `introduction` 字段编辑
- 不支持 `descriptionIndex` 参数

项目工作区版 (`novel-promotion/.../CharacterEditModal.tsx`, 64 行):
- 薄包装，仅做 `appearanceId` 类型转换 (number → string)
- 委托给共享组件 `mode="project"`

**建议**: Asset Hub 版改为 ~50 行的薄包装，传入 `mode="asset-hub"`。

### 5.3 CharacterCard 的架构差异

| 维度 | Asset Hub 版 (520 行) | 项目工作区版 (491 行) |
|------|----------------------|---------------------|
| 架构 | 单体组件，所有逻辑内联 | 模块化拆分为 Header/Gallery/Actions 子组件 |
| Mutation | 内部直接调用 mutation hooks | 通过 props 回调委托给父级 `useCharacterActions` |
| 图片选择确认 | 合并为一个 mutation (`useSelectCharacterImage` + `confirm` flag) | 选择和确认是两个独立 mutation |
| 任务状态 | 通过 `imageTaskRunning` 字段判断 | 通过 `activeTaskKeys` Set 判断 |

**建议**: 抽取共享的图片选择/确认/撤回逻辑为 headless hook，两个版本共用核心逻辑。

### 5.4 API 路由冗余

资产操作存在两套 API 路由:
- **旧版**: `src/app/api/asset-hub/select-image/route.ts` — 通过 `type` + `id` 字段路由
- **新版**: `src/app/api/assets/[assetId]/select-render/route.ts` — 通过 `scope` + `kind` 字段路由

两者最终都调用 `selectAssetRender()`，但入参格式不同。Asset Hub 的 mutation hooks 已迁移到新版 API，旧版路由可能是死代码。
