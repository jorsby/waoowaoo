## 四、数据库 Schema 问题

### 4.1 全局/项目资产双表冗余

当前 Prisma schema 中存在两套平行的资产模型：

| 全局资产 | 项目资产 | 字段重复率 |
|----------|----------|-----------|
| `GlobalCharacter` | `NovelPromotionCharacter` | ~90% |
| `GlobalCharacterAppearance` | `CharacterAppearance` | ~85% |
| `GlobalLocation` | `NovelPromotionLocation` | ~90% |
| `GlobalLocationImage` | `LocationImage` | ~85% |

**关键差异**:
- `NovelPromotionCharacter` 有 `introduction` 字段，`GlobalCharacter` 没有
- `NovelPromotionCharacter` 有 `sourceGlobalCharacterId` 追溯来源
- `GlobalCharacter` 有 `globalVoiceId` 绑定全局音色
- `GlobalCharacterAppearance` 有 `artStyle` 字段，`CharacterAppearance` 没有

**建议**: 短期补齐 `GlobalCharacter.introduction`；长期统一为 `Asset` 基座表 + `scope` 字段区分全局/项目。

### 4.2 `handleConfirmProfile` 破坏性删除

`src/lib/workers/handlers/character-profile.ts:159` 仍然在事务中先 `deleteMany` 所有 `CharacterAppearance`，再重新 `create`。如果分镜面板 (`NovelPromotionPanel`) 引用了被删除的 appearance 的图片 URL，会导致引用失效。

**建议**: 改为 `upsert` 模式，保留已有 appearance 的 ID 和关联关系。

### 4.3 `description` vs `descriptions` 语义模糊

`CharacterAppearance` 和 `GlobalCharacterAppearance` 同时存在：
- `description: String?` — 单个描述
- `descriptions: String?` — JSON 数组格式的多个描述

两者需要手动同步，`description` 通常是 `descriptions[selectedIndex]` 的冗余副本。

## 五、资产中心 (Asset Hub) 深度分析

### 5.1 角色参考图模式对 FAL 的多余依赖

**现状**: 角色图片生成支持"参考图"模式——生成次要形象时自动将主形象图片作为参考，或用户手动上传参考图进行 AI 修图。

**FAL 的特殊耦合** (`src/lib/generators/fal.ts:93-96`):
```typescript
const hasReferenceImages = referenceImages.length > 0
const endpointConfig = FAL_IMAGE_ENDPOINTS[optModelId] || FAL_IMAGE_ENDPOINTS['banana']
const endpoint = hasReferenceImages ? endpointConfig.edit : endpointConfig.base
```

FAL 是唯一需要根据是否有参考图切换 API endpoint 的提供商（base → edit）。其他提供商的处理方式：
- **ARK**: 将参考图转为 Base64，作为 `role: 'reference_image'` 的 content part 传入，无需切换 endpoint
- **Google Gemini**: 直接接受 inline data 格式的参考图
- **OpenAI Compatible**: 有参考图时调用 `client.images.edit()`，无参考图时调用 `client.images.generate()`

**问题**:
1. FAL 的 Data URL 转换逻辑 (`fal.ts:127-145`) 是 FAL 专属的，但写在了通用的生成器文件中
2. 参考图的规范化 (`normalizeReferenceImagesForGeneration`) 在 `outbound-image.ts` 中是通用的，但各 provider 的消费方式完全不同，没有统一的抽象层
3. 如果用户选择了非 FAL 的图片模型，参考图功能的行为取决于该 provider 是否实现了参考图支持，但 UI 层没有对此做能力检测

**建议**: 在 `model-gateway` 或 `AssetKindRegistry` 中注册 `supportsReferenceImage` 能力标记，UI 层根据当前模型的能力决定是否展示参考图上传入口。将 FAL 的 endpoint 切换逻辑封装在 `providers/fal/` 内部，对外暴露统一的 `generate(prompt, options)` 接口。

### 5.3 资产中心图片选择确认后 UI 不刷新 (缓存 Bug)

**复现**: 在资产中心生成多张角色参考图后，点击"确认"选择其中一张，UI 不更新，需要刷新整个页面才能看到结果。

**根因分析**:

确认操作调用链：
```
CharacterCard.handleConfirmSelection()
  → selectImage.mutate({ confirm: true })
    → POST /api/assets/{id}/select-render
      → selectGlobalAssetRender()  // 后端
```

后端确认时的操作 (`src/lib/assets/services/asset-actions.ts:536-562`):
1. 删除所有非选中图片的存储对象
2. 将 `imageUrls` 合并为仅包含选中图片的单元素数组
3. 将 `selectedIndex` 重置为 `0`
4. 将 `descriptions` 合并为仅包含选中描述的单元素数组

前端乐观更新 (`applyCharacterSelection`, `asset-hub-character-mutations.ts:34-59`) 仅更新：
- `selectedIndex` → 设为用户点击的 index
- `imageUrl` → 设为对应 URL

**不匹配**: 乐观更新后缓存中 `imageUrls` 仍是多元素数组、`selectedIndex` 是用户选择的原始 index（如 2），而后端已将 `imageUrls` 缩减为单元素、`selectedIndex` 重置为 0。

`onSettled` 回调 (line 241-244) 确实在 `confirm === true` 时调用了 `invalidateCharacters()`，但存在问题：
- `void invalidateCharacters()` 是 fire-and-forget，错误被静默吞掉
- 乐观更新的缓存数据与服务端返回的数据结构差异过大，React Query 的 reconciliation 可能未能正确覆盖

**对比项目工作区**: 项目工作区使用**独立的确认 mutation** (`useConfirmProjectCharacterSelection`)，其 `onSettled` **无条件**调用 `invalidateProjectAssets()`，不依赖 `confirm` 标志位判断，且不做乐观更新——直接等服务端响应后刷新。

**修复建议**:
1. `applyCharacterSelection` 在 `confirm === true` 时应模拟后端行为：将 `imageUrls` 缩减为单元素、`selectedIndex` 设为 0
2. 或者更简单：confirm 操作不做乐观更新，直接等 `invalidateCharacters()` 的 refetch 结果
3. 将 `void invalidateCharacters()` 改为 `await invalidateCharacters()`，确保 refetch 完成

## 九、模块化拆分方案 (目标架构)

### 9.1 目标目录结构

```
src/
├── app/                    # Next.js 路由层 (页面 + API routes) — 保持不变
│
├── modules/                # 按领域组织的业务模块
│   ├── providers/          # AI 提供商接入层 (ark/, fal/, google/, bailian/, siliconflow/, ...)
│   ├── media-gen/          # 媒体生成引擎 (image/, video/, audio/, lipsync/)
│   ├── llm/                # 大模型推理运行时 (core/, gateway/, observe/, capabilities/, pricing/)
│   ├── task-system/        # 任务调度系统 (core/, workers/, handlers/{analysis,shot-ai,image-gen,...}, sse/)
│   ├── content-pipeline/   # 内容生产流水线 (novel-promotion/, workflow/)
│   └── asset-mgmt/         # 资产管理 (core/, hub/)
│
├── infra/                  # 平台基础设施
│   ├── auth/               # 认证
│   ├── billing/            # 计费
│   ├── db/                 # prisma + redis
│   ├── storage/            # 对象存储
│   ├── logging/            # 日志
│   ├── errors/             # 统一错误处理
│   ├── config/             # 配置中心
│   └── api/                # apiHandler + fetch + rate-limit
│
├── ui/                     # 前端 UI 层
│   ├── components/         # 通用 UI 组件
│   ├── hooks/              # 统一 React Hooks (query/, mutations/, common/)
│   ├── contexts/           # React Context
│   └── styles/             # CSS
│
├── features/               # 自洽的 feature 模块
│   └── video-editor/       # (已有)
│
└── shared/                 # 跨领域共享工具
    ├── types/
    ├── constants.ts
    └── utils/
```

### 9.2 依赖规则

```
app/  ──→  modules/  ──→  infra/
  │           │              │
  └──→  ui/  ─┘              │
          │                  │
          └──→  shared/ ←────┘

规则:
1. app/ 可导入 modules/ 和 ui/
2. modules/ 可导入 infra/ 和 shared/，模块间可有受限依赖
3. ui/ 可导入 modules/ 的类型和 shared/
4. infra/ 只能导入 shared/
5. shared/ 不导入任何上层模块
```

### 9.3 渐进式迁移路径

**Phase 1 (最小侵入)**: 创建 `infra/` 和 `modules/providers/`，移动 41 个散落文件，只改 import 路径不改业务逻辑，删除死代码，添加 tsconfig paths 别名。

**Phase 2 (handlers 分组 + hooks 迁移)**: `workers/handlers/` 按功能拆子目录，`query/hooks/` 和 `mutations/` 搬到 `ui/hooks/`，`components/` 散落文件归类。

**Phase 3 (Provider 收拢)**: 每个 provider 建立完整子目录，`async-poll.ts` 瘦身为路由分发，合并 `api-config` + `config-service`，拆分 `api-errors.ts`。

**Phase 4 (Feature 化)**: 评估将 `novel-promotion` 和 `asset-mgmt` 升级为 feature 模块，补充守卫脚本确保架构边界。
